## Framework & Configuration

The suite runs under **Vitest 3.2.4** on Node.js v25.2.1. Configuration (`vitest.config.ts`):

- `testTimeout: 600000` — 10-minute ceiling per test (Monte Carlo tests run ~50 s each)
- `pool: 'threads'`, `maxThreads: 1` — sequential execution to prevent IPC timeouts
- `sequence.sequent: true` — test files run one at a time

Test files are numbered `test/spec/01_kat.test.ts` through `test/spec/21_x25519.test.ts` (23 files total).

## CI/CD

The full test suite runs automatically on every push to `main` and every pull request via the `test-suite.yml` workflow. **All tests must pass before a PR can be merged.**

The pipeline is structured as two parallel prerequisite jobs followed by five parallel test jobs:

```
                 ┌─→ unit-fast
build ──────────→├─→ unit-montecarlo-cbc
verify-vectors ─→├─→ unit-montecarlo-ebc
                 ├─→ unit-nessie
                 └─→ unit-crypto
```

| Job | Description | Duration |
|-----|-------------|----------|
| `verify-vectors / hashsums` | SHA-256 integrity check of all vector corpus files | ~6s |
| `build / build` | TypeScript compilation | ~9s |
| `unit-fast / tests` | KAT, intermediate, modes, CTR, crypto primitives | ~17s |
| `unit-nessie / tests` | NESSIE Serpent-256 and Serpent-128 vector sets | ~13s |
| `unit-montecarlo-cbc / tests` | Monte Carlo CBC (19.2M iterations) | ~3m 5s |
| `unit-montecarlo-ebc / tests` | Monte Carlo ECB (19.2M iterations) | ~4m 23s |
| `unit-crypto / tests` | Additional cryptographic primitive tests | ~16s |

Total execution time: ~4m 38s (Monte Carlo jobs run in parallel).

### Vector Integrity Verification

Before any test job runs, `verify-vectors` checksums every file in the vector corpus using SHA-256 and compares against stored digests. This ensures the test vectors themselves haven't been accidentally corrupted or modified — testing the tests, so to speak. The build and test jobs only proceed if this check passes.

Push events to `main` skip runs triggered by documentation-only changes (`docs/**`, `*.md`, `LICENSE`, `.gitignore`, `.npmignore`).

### 09 — CTR Mode Vector Tests

File: `test/spec/09_ctr_vectors.test.ts` — **17/17 PASS**

CTR mode predates NIST SP 800-38A and was not included in the original AES submission, so no official Serpent-CTR vectors exist in any corpus. Authoritative vectors are derived using an independent C harness built on top of the verified AES-submission reference implementation.

**Harness**: `sources/first_release_c_and_java/serpent/floppy1/ctr_harness.c`

The Ross Anderson reference implementation (floppy1) was chosen as the ECB base — the same reference that produced the authoritative floppy4 test vectors. `floppy1` uses identical byte ordering to leviathan, requiring no conversion in the harness. This was confirmed by:

```
blockEncrypt(all-zero 256-bit key, all-zero block) = 8910494504181950f98dd998a82b6749
leviathan.encrypt(all-zero 256-bit key, all-zero block) = 8910494504181950f98dd998a82b6749
```

**CTR construction** (identical to `leviathan/src/blockmode.ts` CTR class):
1. Counter initialised as exact copy of IV bytes
2. Keystream_b = `blockEncrypt(key, ctr_b)`
3. ct[i] = keystream[i] XOR pt[i]
4. Counter increment: `ctr[0]++`; carry propagates `ctr[0]→ctr[1]→...→ctr[15]` (little-endian)

**Test cases:**

```
KEY      = 00112233445566778899aabbccddeeff
LONG_KEY = 0000000000000000000000000000000100112233445566778899aabbccddeeff
PT       = 0123456789abcdeffedcba9876543210
CT       = 929dd890dcc881c9a7d8b94b0aa0bad5
```

The `§MANDATORY subkey schedule` test compares all 33 subkeys (SK[0]..SK[32]) from `ecb_iv.txt` against leviathan's derived subkeys after `getSubkeys(key)`.

**Parsing note**: `serpent-aux.c`'s `render()` function prints words from index `size-1` down to `0`, so `SK[i]` in the file is displayed as `X3|X2|X1|X0`. leviathan stores `[X0, X1, X2, X3]` at `this.key[4*i .. 4*i+3]`. The comparison reverses the file's word order before comparing.

### Why R[i] round-state comparison is not done

`ecb_iv.txt` R[i] values are produced by the reference C implementation using **KHat** (SK^[], conventional subkeys via IP permutation), while leviathan uses **K** (SK[], bitslice subkeys loaded via reversed-byte LE format). Both implement the same abstract cipher and produce identical ciphertexts, but their per-round internal states differ.

A direct R[i] comparison would require applying a complex bit-level transformation between leviathan's representation and the reference's IP-based representation. Since the key schedule is verified via SK[], and the final ciphertext is verified for all three key sizes, this provides complete correctness assurance without the invalid R[i] comparison.

## Notes on Implementation Correctness

### Monte Carlo key update formula

The chain-continuity test for ECB Monte Carlo verifies the key update formula between outer iterations. The correct formula (empirically verified against `ecb_e_m.txt`) is:

```
concat = CT_9999 || CT_9998    (LAST output first)
suffix = FIRST keyBytes of concat
```

For each key size this gives:
- **128-bit**: suffix = CT_9999 (16 bytes)
- **192-bit**: suffix = CT_9999 ‖ CT_9998[0..7] (24 bytes)
- **256-bit**: suffix = CT_9999 ‖ CT_9998 (32 bytes)

**Verification**: For KEYSIZE=192, I=0 (all-zero key), CT_9999 = `2d8af7b79eb7f21fdb394c77c3fb8c3a`. I=1 KEY starts with `2d8af7b79eb7f21fdb394c77c3fb8c3a` (= CT_9999 XOR 0 key) — confirmed. The chain continuity test passes for all 400 × 3 = 1200 entries.

### NESSIE byte-order preprocessing

NESSIE vectors use standard big-endian byte order. leviathan uses AES-submission byte order (bytes are reversed before being packed into 32-bit little-endian words). The correct preprocessing is to **reverse all bytes** of the key, plaintext, and ciphertext.

This was determined empirically by testing all combinations of byte/word transformations against multiple known-good vectors. The discovery process and the discrepancy with the developer's note (which describes preprocessing for a different C reference implementation) are documented in `test/helpers/nessie.ts`. The same full-byte-reversal transform applies correctly to all key sizes.
