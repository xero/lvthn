## Philosophy

### Why test vectors matter for cryptographic libraries

Cryptographic code is uniquely difficult to validate because incorrect output is
indistinguishable from correct output by inspection. A block cipher that produces
the wrong ciphertext does not crash, throw exceptions, or fail any generic
software quality check. It simply produces bytes — and those bytes look
exactly as random and well-structured as the correct bytes would.

This is why test vectors from authoritative external sources are the only
meaningful correctness signal for a cryptographic implementation. The vectors
are not merely regression tests that detect regressions against the same code
— they are ground truth from an independent party who implemented or defined
the algorithm independently. When leviathan's Serpent produces the same output
as the Ross Anderson / Eli Biham / Lars Knudsen reference implementation for
every one of 2,496 KAT vectors and every iteration of 19.2 million Monte Carlo
inner loops, that is evidence that the implementations are describing the same
cipher.

Coverage metrics are meaningless for crypto. One hundred percent line coverage
of a Serpent implementation with a wrong S-box is worse than fifty percent
coverage of a correct one, because the hundred-percent case provides false
confidence. A test suite that generates its own expected values from the same
implementation it is testing — what is sometimes called "round-trip regression
without ground truth" — cannot detect systematic errors. The SHA-256 audit
documented in this repository is a concrete example of how an agent can
generate a convincing passing test suite against wrong vectors, and why external
authoritative vectors are non-negotiable.

### What "verified" means in this library

This document uses three tiers throughout:

**VERIFIED**: The value has been confirmed against at least two independent
references. For standards-defined primitives this means the value matches both
the published standard document and at least one independent implementation
(typically OpenSSL and/or Python hashlib/hmac). For NESSIE vectors, "verified"
in the filename means the NESSIE project confirmed these values against multiple
independent Serpent implementations during the competition evaluation period.

**PARTIAL**: Some values in the set have been verified against independent
references; the remainder serve as a regression suite. This tier is used when
verifying every vector in a large set would require significant work but the
verified subset provides meaningful assurance.

**UNVERIFIED**: The values came with the implementation or from a secondary
source and have not been independently checked against the primary standard
document. These vectors detect regressions against the current implementation
but do not independently prove correctness.

**GENERATED**: No external authoritative source exists for the construction
being tested. Values were self-generated by a harness or reference tool and
cross-checked internally. This tier is used for Serpent-CTR: the AES submission
predates CTR mode standardization, and no third-party CTR vectors for Serpent
exist anywhere. The generated vectors establish internal consistency but cannot
be verified against an independent authority.

### The principle: never change a vector to match implementation output

When an authoritative vector and an implementation disagree, the implementation
is wrong, not the vector. This principle sounds obvious but is surprisingly
easy to violate in practice.

The SHA-256 audit (documented in detail in `SHA256_audit.md`) provides a
concrete cautionary example. An agent tasked with diagnosing a failing SHA-256
test observed that OpenSSL and Python produced output consistent with the
implementation, correctly identified that the implementation agreed with
external tools, but then drew the wrong conclusion: it changed the FIPS 180-4
test vector to match the implementation. This produced a passing test suite
built on a wrong expected value. The error was a typo in the task file that
provided the vector, the FIPS document and OpenSSL agree; the task file did
not. A passing test suite built on a wrong vector is worse than a failing test
suite, because it provides false confidence while silently hiding the error.

**The cardinal rule: never change a vector value to make a test pass. If a
vector fails, debug the implementation or re-verify the vector against the
primary source document. Never modify the expected value.**

## Test Infrastructure

### Test runner

- **Framework**: Vitest 3.2.4
- **Runtime**: Node.js v25.2.1
- **Config**: `vitest.config.ts`
- **Command**: `npx vitest run`
- **Alias**: `npm test`

### Configuration

```typescript
// vitest.config.ts
{
  testTimeout: 600000,   // 10 minutes: Monte Carlo tests run ~50 s each
  pool: 'threads',
  poolOptions: { threads: { maxThreads: 1, minThreads: 1 } },
  sequence: { sequent: true },   // test files run one at a time
}
```

Sequential single-threaded execution is required because the Monte Carlo tests
are CPU-intensive. Running them in parallel would cause IPC timeouts and
spurious failures.

### Build process

leviathan is TypeScript. Vitest imports from `src/` directly via ts-node.
No compilation step is required to run the test suite. The `dist/` directory
exists for distribution purposes and is not used during testing.

### Serpent: AES Submission Vectors

**Source**: Serpent AES candidate submission by Ross Anderson, Eli Biham, Lars Knudsen
**Submission page**: https://www.cl.cam.ac.uk/~rja14/serpent.html
**Archive**: http://www.cl.cam.ac.uk/~rja14/Papers/serpent.tar.gz
**Status**: VERIFIED

These are the reference vectors from the original AES competition submission. They cover
128-bit, 192-bit, and 256-bit key sizes. The files in `test/vectors/` are direct copies
from `sources/first_release_c_and_java/serpent/floppy4/`, the original submission archive.

#### Vector files

**`ecb_vt.txt`** — Variable-text KAT (384 vectors)
Fixed all-zero key per key size; plaintext sweeps all one-hot 128-bit patterns.
Tests: encrypt and decrypt, all 384 vectors.

**`ecb_vk.txt`** — Variable-key KAT (576 vectors)
Fixed all-zero plaintext; key sweeps all one-hot patterns across 128/192/256-bit sizes.
Tests: encrypt and decrypt, all 576 vectors.

**`ecb_tbl.txt`** — S-box table entry tests (1,536 vectors)
Exhaustive S-box isolation: targets specific S-box inputs to detect table lookup errors.
These are the most targeted vectors in the KAT suite — a single wrong S-box table entry
will cause a failure here before it would appear in ecb_vt or ecb_vk.

**`ecb_iv.txt`** — Intermediate value tests (3 test cases)
Contains the full subkey schedule (SK[0..32]) and round state (R[0..31]) for each key
size. These are the most diagnostic vectors in the entire suite — a key schedule bug
will show up immediately in the SK comparison even if the final ciphertext is correct.

Note: The R[i] round state values are not compared. The reference C implementation
(floppy1) uses the KHat representation (IP permutation applied), while leviathan uses
the K bitslice representation. Both produce identical final ciphertexts but their
per-round internal states differ. Correctness is established by SK[0..32] match (key
schedule) plus final CT match for all three key sizes (cipher function).

**`ecb_e_m.txt` / `ecb_d_m.txt`** — Monte Carlo ECB (1,200 outer × 10,000 inner)

**`cbc_e_m.txt` / `cbc_d_m.txt`** — Monte Carlo CBC (1,200 outer × 10,000 inner)

#### The Monte Carlo tests

Each of 1,200 outer iterations specifies a starting key and plaintext; the inner loop
runs 10,000 encrypt (or decrypt) operations, feeding each output back as the next input.
A single wrong bit in the block function — S-boxes, linear transform, or key schedule
— will produce a completely wrong output after 10,000 inner iterations. The error
compounds rather than averaging out. The 1,200 vectors cover all three key sizes.
The ECB and CBC variants together represent approximately 19.2 million cipher operations.

**Key update formula**:

```
concat = CT_9999 ‖ CT_9998   (last output first)
new_key = XOR(old_key, first keyBytes of concat)
```

The order of concatenation is `CT_9999 ‖ CT_9998` — most recent output first. The
alternate ordering (`CT_9998 ‖ CT_9999`) produces wrong results. This was discovered
empirically during Phase 4 of the audit. See `test_suite.md` for the verification trace.

### Serpent — NESSIE Vectors

**Source**: NESSIE project (New European Schemes for Signatures, Integrity, and Encryption)
**Project page**: https://www.cosic.esat.kuleuven.be/nessie/
**Vectors page**: https://www.cosic.esat.kuleuven.be/nessie/testvectors/
**Status**: VERIFIED

The NESSIE project evaluated candidate algorithms for European cryptography
standardization (2000–2003). "Verified" in the filename means the NESSIE project
confirmed these values against multiple independent Serpent implementations.

Two vector sets are included:
- **Serpent-256-128**: 1,284 vectors across 8 sets (256-bit key, 128-bit block)
- **Serpent-128-128**: 1,028 vectors across 8 sets (128-bit key, 128-bit block)

Each vector is tested for both encrypt and decrypt (2,568 and 2,058 test cases). All pass.

#### NESSIE byte-order preprocessing

NESSIE uses big-endian byte order. leviathan uses AES-submission byte order (bytes
reversed before packing into 32-bit little-endian words). All NESSIE keys, plaintexts,
and ciphertexts require full byte-reversal before use with leviathan. This was determined
empirically — there is no documentation of the discrepancy. Implementation:
`test/helpers/nessie.ts`. Unit tests: `06_nessie_helpers.test.ts`.

### Serpent — CTR Mode Vectors (GENERATED)

**Source**: [floppy1/ctr_harness.c](https://github.com/xero/lvthn/blob/floppy1/ctr_harness.c)
**Status**: GENERATED

No authoritative external CTR vectors exist for Serpent. The AES submission predates
CTR mode standardization (NIST SP 800-38A, 2001).

The 5 test cases in `09_ctr_vectors.test.ts` were generated by a C harness built on the
floppy1 Ross Anderson reference implementation (same byte ordering as leviathan — no
byte-order conversion needed in the harness).

**Portability note**: `typedef unsigned long WORD` in floppy1 is 8 bytes on arm64 macOS.
The harness uses explicit `bytes_to_block`/`block_to_bytes` helpers.

**CTR construction** (identical to `src/blockmode.ts` CTR class):
1. Counter initialised as exact copy of IV bytes
2. Keystream block b = blockEncrypt(key, counter_b)
3. ct[i] = keystream[i] XOR pt[i]
4. Counter increment: byte 0 first; carry propagates 0→1→…→15 (little-endian 128-bit)

**Cross-check**: Block 0 keystream for Cases A (128-bit zero key) and B (256-bit zero
key) verified to equal leviathan ECB encrypt of all-zero counter block.

**Limitation**: These vectors establish internal consistency between leviathan and floppy1
but cannot be verified against an independent third-party source.

## Process: Adding New Test Vectors

### Before adding vectors

1. Identify the authoritative source document (NIST publication, RFC, IETF draft).
2. Download the document directly from the primary source — do not copy values from
   secondary sources, blog posts, or existing implementations.
3. Record the document URL and publication date.
4. Note which sections contain the specific test cases you need.

### Verifying vector values

Never add a vector without independently verifying its value against at least two
independent tools:

```bash
# SHA-256
echo -n "your input" | openssl sha256
python3 -c "import hashlib; print(hashlib.sha256(b'your input').hexdigest())"

# SHA-512
echo -n "your input" | openssl sha512
python3 -c "import hashlib; print(hashlib.sha512(b'your input').hexdigest())"

# HMAC-SHA-256
python3 -c "
import hmac, hashlib
key = bytes.fromhex('your_key_hex')
msg = b'your message'
print(hmac.new(key, msg, hashlib.sha256).hexdigest())
"

# PBKDF2-SHA256
python3 -c "
import hashlib
dk = hashlib.pbkdf2_hmac('sha256', b'password', b'salt', 1)
print(dk.hex())
"
```

If two independent tools disagree, stop. Investigate before adding any vector.

### The cardinal rule

**Never change an expected value to make a test pass.**

If an implementation produces output that differs from an authoritative vector, the
implementation is wrong. The vector is ground truth. Debug the implementation — not
the vector.

The only exception: if you have independently verified via multiple tools that the
vector value in the test file is a typo and does not match the primary source document,
correct the vector value and document the correction. This is a vector correction, not
a vector-to-implementation accommodation.

See `SHA256_serpent_audit.md` for a concrete case where this principle was violated and
subsequently corrected.

### The provenance block format

Every vector file must have a provenance block at the top:

```typescript
/**
 * Test vectors — [primitive name]
 *
 * Source: [full citation]
 * URL:    [direct link to document or page]
 * Date:   [publication date of the standard]
 *
 * Coverage: [which sections / test cases are included]
 * Not covered: [which are absent and why]
 *
 * Verification: [how values were independently confirmed]
 *
 * Audit status: VERIFIED | PARTIAL | UNVERIFIED | GENERATED
 */
```

For GENERATED vectors, add:

```typescript
 * Note: No authoritative external vectors exist for this construction.
 * Generated by: [describe harness/tool]
 * Cross-checked: [describe verification method]
 * Limitation: establishes internal consistency only — no third-party authority.
```

### Vector file vs inline vectors

- Large corpora (>10 vectors): use a separate file in `test/spec/` (e.g. `sha256_vectors.ts`)
- Small reference sets (≤10 vectors from a specific standard): may be inline in the test
  file with per-vector source comments
- Self-generated vectors: always in a dedicated file clearly marked GENERATED

### Updating existing provenance blocks

If you add vectors to an existing file, update the provenance block to include the new
source, update the count, and update the audit status if warranted.

## Test Execution Order and Diagnostic Value

When debugging a failure, run lower-numbered files first — each file tests a more
fundamental property than the next.

| File | What a failure means |
|------|---------------------|
| `01_kat.test.ts` | Block function wrong (S-boxes, linear transform, or key schedule). All other Serpent tests will also fail. |
| `02_intermediate.test.ts` | Key schedule bug — subkey derivation is wrong independent of the block function. |
| `03_monte_carlo_ecb.test.ts` | Extremely sensitive: a single wrong bit compounds over 10,000 iterations. |
| `04_monte_carlo_cbc.test.ts` | CBC mode wrapper bug, or block function bug not caught by ECB Monte Carlo. |
| `05_serpent_modes.test.ts` | Mode wrapper (CTR/CBC/PKCS7) logic error. |
| `06_nessie_helpers.test.ts` | NESSIE preprocessing (byte-reversal) logic wrong. All NESSIE tests will also fail. |
| `07/08_nessie_*.test.ts` | Implementation wrong for the given key size, or preprocessing wrong. |
| `09_ctr_vectors.test.ts` | CTR counter construction or increment wrong. |
| `17_sha256.test.ts` | SHA-256 algorithm wrong — check FIPS 180-4 vectors first (they are authoritative). |
| `13_hmac.test.ts` | HMAC construction wrong, or SHA-256 wrong. Run 17 first. |

## Lessons Learned

### The SHA-256 audit (2026-02-28)

*Full record: `SHA256_serpent_audit.md`*

A test was added with FIPS 180-4 vectors for SHA-256. The implementation produced output
that didn't match the FIPS vector for `"abc"`. An agent tasked with diagnosing the bug
ran OpenSSL and Python to verify SHA-256("abc") and correctly found they agreed with the
implementation output.

The agent then drew the wrong conclusion: rather than questioning whether the FIPS vector
in the task file was correct, it concluded the FIPS vector itself was wrong and changed
the test vector to match the implementation. This produced a passing test suite built on
a wrong expected value.

The implementation was never broken. The error was a typo in the task file: the provided
"abc" expected value (`...2ec73b00361bbef0...`) was wrong. The correct FIPS value
(`...2223b00361a396...`) matches OpenSSL, Python, and the leviathan implementation.

**Key lesson**: When implementation and vector disagree, always re-verify the vector
against the primary source document before assuming either side is wrong. Task files and
prompts can contain typos; FIPS publications do not. Changing a vector to match
implementation output is always wrong — the correct response is to find and fix the typo.

### The NESSIE byte-order discovery

NESSIE vectors use big-endian byte order while leviathan uses AES-submission byte order
(from floppy4). No documentation of this discrepancy existed. The correct preprocessing
— full byte reversal of key, plaintext, and ciphertext — was determined by empirically
testing all candidate transformations against known-good ECB outputs. The developer's
note on the NESSIE website described preprocessing for a different C reference
implementation, not for leviathan.

**Key lesson**: Vector file format assumptions must be verified, not assumed. Two correct
implementations of the same cipher can use different byte orderings and produce
completely different-looking hex strings for the same input.

### The Monte Carlo key update formula

The Monte Carlo key update concatenates the last two ciphertexts. The correct order is
`CT_9999 ‖ CT_9998` (most recent first). The alternate ordering produces wrong results
and is not obviously wrong from reading the specification text. This was discovered by
comparing against the floppy4 reference vectors.

**Key lesson**: Monte Carlo algorithm details are not fully specified in the vector
files. When implementing a Monte Carlo test, verify the key update formula empirically
against the reference before trusting the results.

### The CTR counter format

The CTR counter is a 128-bit little-endian integer: byte 0 is least significant; carry
propagates 0→1→…→15. This matches floppy1's behavior. The counter format was verified
by cross-checking CTR output against ECB encryption of explicit counter values.

**Key lesson**: "CTR mode" does not fully specify behavior. Counter endianness (big vs
little), counter width (32-bit, 64-bit, 128-bit), and counter position must all be
explicitly documented and verified.
